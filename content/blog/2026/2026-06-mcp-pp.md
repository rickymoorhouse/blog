<!-- Concept -->
Agents are only as good as the access they have to accurate data and business functions. LLMs have a cut off point as to what they know - they're trained on a set of data and anything beyond that is unknown to them - therefore to build useful agents you need to provide a way for them to access data beyond that corpus.  This could be more your own business data or just more recent information.

To enable this Anthropic, in 2024, [introduced Model Context Protocol][anthropic-blog] (MCP) as an open standard to build these connections to data-sources and tooling.  Since then, MCP has exploded into the way of providing tools to AI Agents. It may not be the only way for agents to access data but it has become the most common.

<!--Mechanics -->
The backbone of todays internet is the API based around a defined contract guaranteeing stable predictable, consistent behaviour which is what production systems require.  Whilst REST APIs are designed for stability, MCP is designed for agility.

- **REST APIs** are published with a static contract. Consumers know exactly how they will work and if this is to change there is typically a new version and a period to move to it.  
- **GraphQL APIs** provide an introspectable schema so the client can see what is available and request the specific data fields needed. This provides a level of insulation between the backend system architecture and clients requirements.  
- **MCP Servers** provide a list of available tool specifications to the agent which are then pre-populated into the context window each session by making a list tools call. This discovery means that the provider can completely change the tool definitions without notice - and they agent will have the updated definitions to select from. It then uses the tools from will also adapt to use the tool that matches the current task.

This difference can have pros and cons - for example clients can depend on an API to respond according to the providers contract.  Where as with MCP there is no long lasting contract beyond the session - this enables the provider to adjust the tools they provide without notification or versioning cycle. 


<!-- Real-world behavior -->
If things fail, the agent will still try and work around the problem - for example whilst working on an MCP tool to lookup tenant details and which tier they were configured to, my agent found the equivalent OpenAPI specification locally and tried to combine the two. When the Agent gets a failure response back from a tool as part of the review it determines the task is not complete and considers other routes to complete it. This could include other MCP tools but in this case it was reading local files to identify a way to complete the task. 

<!-- Risks -->
There are thousands of pre-built MCP servers out there - some official but mostly unofficial community built servers created by interested developers.  According to [Equixly] 43% of MCP servers contain command injection flaws and 30% permit unrestricted URL fetching, so it is important to understand what you are running and to have control over it. <!-- intensify with clear narrative and managing trust in external dynamic tooling -->

This highlights the fact that traditional security practices still apply to the agentic space - often more than ever.  Supply chain security and least privilege are more critical than ever when you have non-deterministic agent flows handling data through tooling that comes from the community.

<!-- Responsibility. -->

Ultimately, the responsibility for the security and integrity of these systems remains entirely with the developer. Because agentic flows are non-deterministic the risk of an unintended action, triggered by a flawed community tool or a prompt injection, is higher than in traditional software. 

## Security checklist for MCP

- Audit the source - before you connect a community MCP server to your production systems, review the code manually.
- Isolate - isolate the server to a container to limit the risk of command injection.
- Human-in-the-loop - Ensure that sensitive actions require a human approval that is made with all the context available.
